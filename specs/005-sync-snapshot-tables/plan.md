# Implementation Plan: Sync Snapshot Tables Locally

**Branch**: `005-sync-snapshot-tables` | **Date**: 2026-02-19 | **Spec**:
[spec.md](file:///e:/Work/My%20Projects/Monyvi/specs/005-sync-snapshot-tables/spec.md)

Feature specification from `/specs/005-sync-snapshot-tables/spec.md`

## Summary

Bring the three daily snapshot tables (`daily_snapshot_balance`,
`daily_snapshot_assets`, `daily_snapshot_net_worth`) into local WatermelonDB
sync. Drop the `breakdown` JSONB column from balance and assets tables. Re-add
missing UNIQUE constraints. Replace the API-based net worth comparison with a
local query. Remove the API endpoint and mobile service file.

## Technical Context

**Language/Version**: TypeScript (strict mode)  
**Primary Dependencies**: WatermelonDB, Supabase, React Native/Expo  
**Storage**: PostgreSQL (Supabase) + SQLite (WatermelonDB)  
**Testing**: Manual E2E (Maestro or manual), existing unit tests  
**Target Platform**: iOS & Android via Expo  
**Project Type**: Mobile + API monorepo  
**Constraints**: Offline-capable, 90-day data retention

## Implementation Phases

### Phase 1: Database Migration

> [!IMPORTANT] Per project rules, all DDL changes go through local migration
> files. Never use Supabase MCP for DDL.

#### [NEW] [025_sync_snapshot_tables.sql](file:///e:/Work/My%20Projects/Monyvi/supabase/migrations/025_sync_snapshot_tables.sql)

SQL migration to:

1. Drop `breakdown` JSONB column from `daily_snapshot_balance`
2. Drop `breakdown` JSONB column from `daily_snapshot_assets`
3. Add `UNIQUE(user_id, snapshot_date)` constraint on all 3 tables
4. Update cron functions to match new schema (remove `breakdown` from INSERT,
   fix ON CONFLICT clauses)

---

### Phase 2: WatermelonDB Schema Generation

#### [MODIFY] [transform-schema.js](file:///e:/Work/My%20Projects/Monyvi/scripts/transform-schema.js)

1. Remove `daily_snapshot_balance`, `daily_snapshot_assets`,
   `daily_snapshot_net_worth` from `EXCLUDED_TABLES`
2. Add entries to `TABLE_TO_CLASS` mapping:
   - `daily_snapshot_balance` → `DailySnapshotBalance`
   - `daily_snapshot_assets` → `DailySnapshotAssets`
   - `daily_snapshot_net_worth` → `DailySnapshotNetWorth`

Then run:

- `npm run db:push` — apply migration to remote
- `npx supabase gen types` — regenerate Supabase types
- `npm run db:migrate` — regenerate WatermelonDB schema, models, and local
  migrations

---

### Phase 3: Sync Logic

#### [MODIFY] [sync.ts](file:///e:/Work/My%20Projects/Monyvi/apps/mobile/services/sync.ts)

1. Remove `daily_snapshot_*` from `EXCLUDED_TABLES`
2. Add a new type alias for snapshot table names (read-only, pull-only)
3. Create `pullSnapshotTable()` function:
   - Accepts: table name, userId, daysToKeep (default 90)
   - Queries by `user_id` AND `created_at > cutoffDate`
   - Returns all records as `updated` (like `pullMarketRates`)
   - No deleted handling (read-only tables)
4. Add routing in `pullChanges()` for snapshot tables → `pullSnapshotTable()`
5. Add snapshot tables to the push exclusion logic (they must never be pushed)

---

### Phase 4: Replace API with Local Query

#### [MODIFY] [useNetWorth.ts](file:///e:/Work/My%20Projects/Monyvi/apps/mobile/hooks/useNetWorth.ts)

Rewrite `useMonthlyPercentageChange` to:

1. Query `daily_snapshot_net_worth` from local WatermelonDB
2. Use `getSameDayLastMonth()` to find the comparison date
3. Sort by `snapshot_date DESC`, get the closest snapshot to the comparison date
4. Calculate `percentageChange` locally using the `NetWorthComparison` interface
5. Make it reactive using WatermelonDB's `observe()` pattern (updates
   automatically on sync)
