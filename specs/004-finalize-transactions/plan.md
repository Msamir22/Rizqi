## Contracts

### Extended `updateTransaction` Service

```typescript
interface TransactionUpdatePayload {
  readonly amount?: number;
  readonly categoryId?: string;
  readonly note?: string;
  readonly date?: Date;
  readonly counterparty?: string; // NEW
  readonly type?: TransactionType; // NEW — EXPENSE ↔ INCOME
  readonly accountId?: string; // NEW — cross-currency allowed
}
```

**Balance logic for type change:**

```text
EXPENSE→INCOME: balance += 2 × amount (revert -amount, apply +amount)
INCOME→EXPENSE: balance -= 2 × amount (revert +amount, apply -amount)
```

**Balance logic for account change:**

```text
Old account: revert effect (EXPENSE: +amount, INCOME: -amount)
New account: apply effect  (EXPENSE: -amount, INCOME: +amount)
```

### New `convertTransactionToTransfer` Service

```typescript
interface ConvertToTransferPayload {
  readonly transactionId: string;
  readonly toAccountId: string;
  readonly notes?: string;
}

function convertTransactionToTransfer(
  payload: ConvertToTransferPayload
): Promise<void>;
```

Atomic operation: soft-deletes the transaction, reverts its balance, creates a
new transfer, adjusts both accounts.

### New `convertTransferToTransaction` Service

```typescript
interface ConvertToTransactionPayload {
  readonly transferId: string;
  readonly type: "EXPENSE" | "INCOME";
  readonly categoryId: string;
  readonly accountId: string; // defaults to fromAccountId, user can change
}

function convertTransferToTransaction(
  payload: ConvertToTransactionPayload
): Promise<void>;
```

Atomic operation: soft-deletes the transfer, reverts both account balances,
creates a new transaction, adjusts the single target account.

## Complexity Tracking

No constitution violations. All work stays within existing architecture
patterns.

| Complexity Item                 | Justification                                                              |
| ------------------------------- | -------------------------------------------------------------------------- |
| Type change balance logic       | Net delta of `±2 × amount` — must be tested thoroughly                     |
| Account swap balance logic      | Dual-account revert/apply, cross-currency allowed                          |
| Transaction→Transfer conversion | Cross-table atomic operation — must ensure data integrity                  |
| Transfer→Transaction conversion | Cross-table atomic operation — mirror of Transaction→Transfer              |
| Generic ConfirmationModal       | Must be flexible enough for delete (danger) and discard (warning) variants |
| Trigger removal migration       | Must DROP triggers without breaking existing data                          |
